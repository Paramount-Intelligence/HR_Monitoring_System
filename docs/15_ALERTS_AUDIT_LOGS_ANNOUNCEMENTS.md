# 15 Alerts, Audit Logs, and Announcements

This module provides the governance and visibility layer for the system, ensuring all critical events are captured, communicated, and audited.

## 1. System Alerts
Alerts are automatically generated by the background worker to highlight operational exceptions.
*   **Types**:
    *   `missing_checkout`: Employee checked in but didn't check out within the buffer.
    *   `overdue_task`: Task past its due date.
    *   `idle_after_checkin`: Check-in recorded but no timer/task activity for several hours.
    *   `approval_delay`: Project or Leave request pending for too long.
    *   `workload_overload`: Employee assigned tasks exceeding their capacity.
*   **Severity**: Low, Medium, High, Critical.
*   **Actions**: Alerts can be `resolved` (action taken) or `dismissed` (ignored).
*   **Communication**: Alerts trigger email notifications to the relevant Manager or Admin.

## 2. Audit Logs
The system maintains an immutable trail of sensitive actions to ensure accountability.
*   **Scope**:
    *   **User Management**: Creation, status changes (Suspension/Activation), and Role changes.
    *   **Governance**: Project and Leave approvals/rejections (includes decision, reason, and actor).
    *   **Manual Edits**: Any manual edits to attendance or time logs.
    *   **Security**: Admin bootstrap and permission seeding events.
*   **Data Structure**: Captures `actor_user_id`, `action_type`, `entity_type`, `old_value`, `new_value`, and a timestamp.
*   **Integrity**: Audit logs are read-only and cannot be modified or deleted by any user, including Admins.

## 4. Operational Requirements
*   **Null-Safe Processing**: Audit log and Alert display logic must handle deleted or missing related entities gracefully (e.g., if a project is deleted, the log should still show its title/ID as a string).
*   **No Raw IDs**: User-facing lists must show labels (User Name, Project Title) instead of UUIDs.
*   **PKT Timestamps**: All logs and alerts must display their occurrence time in **PKT**.
